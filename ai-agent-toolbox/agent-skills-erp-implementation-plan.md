# Agent Skills × ERP 智慧層實作方案

> 以聖保羅烘焙花園為案例，規劃 AI Agent 如何作為 ERP 的智慧增強層

---

## 執行摘要

### 核心結論

| 問題 | 答案 |
|------|------|
| Agent Skills 能取代 ERP 嗎？ | **不能**，核心交易處理仍需 ERP |
| Agent Skills 能減輕 ERP 痛點嗎？ | **可以**，輸入、查詢、例外處理大幅改善 |
| 聖保羅應該怎麼做？ | **ERP 為骨，Agent 為肉** — 混合架構 |

### 核心理念：AI 做人的事，ERP 做機器的事

```
┌─────────────────────────────────────────────────────────────────┐
│  傳統 ERP 模式                                                   │
│  ─────────────                                                  │
│  人 → 學習介面 → 精確輸入 → 系統處理 → 查報表 → 理解數據 → 決策   │
│       ↑ 痛苦      ↑ 痛苦              ↑ 痛苦    ↑ 痛苦           │
├─────────────────────────────────────────────────────────────────┤
│  Agent 智慧層模式                                                │
│  ───────────────                                                │
│  人 → 說人話 → Agent 轉譯 → ERP 處理 → Agent 解讀 → 人話回覆 → 決策│
│       ↑ 自然    ↑ AI 負責            ↑ AI 負責    ↑ 自然        │
└─────────────────────────────────────────────────────────────────┘
```

## 一、ERP vs Agent Skills 的定位

### ERP 的本質

ERP 不只是「軟體」，而是三件事的組合：

| 層面 | ERP 提供的 | Agent Skills 能做的 |
|------|-----------|-------------------|
| **資料模型** | 標準化的會計科目、物料主檔、BOM 結構 | ✗ 需要人為設計，無現成標準 |
| **業務邏輯** | MRP 運算、成本分攤、稅務申報 | △ 可實作，但需逐一開發 |
| **流程控制** | 審批流、權限、稽核軌跡 | ✓ 可透過 Skills 實現 |
| **法規合規** | 符合 IFRS、稅務申報格式 | ✗ 需要認證，AI 無法取代 |
| **生態整合** | 銀行、稅務機關、EDI 標準 | △ 可串接，但需逐一處理 |

### Agent Skills 真正的優勢

Agent Skills 不是要「取代」ERP，而是解決 ERP 的三大痛點：

| 痛點 | 傳統 ERP | Agent 解法 |
|------|---------|-----------|
| **輸入地獄** | 15 個欄位精確輸入 | 拍照 / 語音 / 自然語言 |
| **查詢困難** | 登入 → 找報表 → 讀數字 | 問一句話得到答案 |
| **例外處理** | 遇到非標準就卡住 | AI 提供選項、建議處理方案 |

### 場景 1：採購入庫 — 從「輸入地獄」到「拍照搞定」

**傳統 ERP 痛點**：
- 收到貨要手動輸入：供應商、品號、數量、批號、效期、倉位
- 一張入庫單 15 個欄位，平均 3-5 分鐘
- 品號記不住，要查系統
- 打錯字就對不上帳

**Agent Skills 解法**：

```
┌─────────────────────────────────────────────────────────────────┐
│  Skill: 智慧入庫助手                                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  倉管：[拍照] 送貨單                                             │
│                                                                 │
│  Agent：我辨識到這是大成的送貨單                                  │
│         • 低筋麵粉 25kg × 20 包 = 500kg                          │
│         • 批號：L240115                                         │
│         • 效期：2025/01/15                                      │
│         • 建議倉位：A-03-02（同批號旁邊）                         │
│                                                                 │
│         要我建立入庫單嗎？                                       │
│                                                                 │
│  倉管：好，但放 A-03-01                                          │
│                                                                 │
│  Agent：已修改倉位為 A-03-01                                     │
│         入庫單 PO-2026-0115-001 已建立                           │
│         請確認實際數量後點「完成入庫」                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**技術實作**：
```yaml
Skill: smart-receiving
Triggers:
  - 圖片輸入（送貨單、發票）
  - 語音：「收到 XXX 的貨」

Workflow:
  1. OCR 辨識送貨單
  2. 比對 ERP 供應商主檔 → 模糊匹配
  3. 比對 ERP 物料主檔 → 模糊匹配 + 歷史採購
  4. 查詢倉位建議（同批號、FIFO、空間）
  5. 產生入庫單草稿
  6. 等待人確認
  7. 呼叫 ERP API 建單

ERP Integration:
  - GET /api/vendors?keyword={辨識結果}
  - GET /api/materials?keyword={辨識結果}
  - GET /api/warehouse/suggest?material_id={}&qty={}
  - POST /api/receiving-orders
```

**效益比較**：

| 指標 | 傳統 ERP | Agent 智慧層 |
|------|---------|-------------|
| 每單輸入時間 | 3-5 分鐘 | 30 秒 |
| 錯誤率 | 5-8% | < 1%（AI 預填 + 人確認）|
| 訓練時間 | 2 週 | 10 分鐘 |
| 品號查詢次數 | 每單 2-3 次 | 0 次 |

```yaml
Skill: production-advisor
```

## 四、最強大腦觀點

### Marc Andreessen（a16z 創辦人，企業軟體投資教父）

> *"Software is eating the world, but AI is eating software."*
> *"The best enterprise software is invisible. The user shouldn't know they're using software."*

**核心洞察**：

Agent 智慧層做對了一件事：**讓 ERP 變成隱形的後端**。使用者跟 LINE Bot 說話，不知道背後是鼎新還是 SAP。

**但要注意**：不要把 Agent 當成 ERP 的化妝品。如果 Agent 只是把 ERP 的複雜性翻譯成人話，只解決了 10% 的問題。真正的機會是讓 Agent 做 ERP 不敢做的事：**容錯、模糊匹配、常識推理**。

---

### Paul Graham（Y Combinator 創辦人）

> *"Do things that don't scale."*
> *"The most dangerous thing is a grand vision executed without talking to users."*

**核心洞察**：

這份規劃太「工程師思維」了。在想架構、API、Phase 1/2/3 之前，先回答：**聖保羅的人真的痛嗎？**

**建議**：
1. **先不要建系統**。找一個倉管，連續一週站在他旁邊，看他怎麼工作。
2. **手動模擬 Agent**。你自己當那個 AI，用 LINE 回覆他。看他會問什麼、會怎麼問、會犯什麼錯。
3. **找到一個讓他「哇」的瞬間**。當他說「這樣太方便了」，你就知道該從哪裡開始。

然後再來談架構。

---

### Clay Christensen（破壞式創新理論創始人）

> *"Disruption comes from below, not from above."*
> *"The innovator's dilemma is that the best customers are the last to adopt disruptive technology."*

**核心洞察**：

問題框架錯了。「Agent 能不能取代 ERP」不是破壞式創新的運作方式。

**破壞式創新不是取代，是讓「非消費者」變成消費者。**

ERP 的問題不是功能不夠，是門檻太高。一套鼎新 ERP 要多少錢？導入要多久？多少人的公司才用得起？

Agent Skills 的真正機會不是幫聖保羅（14 億營收）省人力，而是讓「原本用不起 ERP 的」小烘焙店也能有基本的進銷存管理。

**你的客戶不是聖保羅，是聖保羅的競爭對手**——那些 5 人、10 人的小店。當他們用 NT$8,000/月的 Agent 方案做到聖保羅 80% 的管理能力，聖保羅才會真正緊張。

---

### Jensen Huang（NVIDIA CEO）

> *"The more you buy, the more you save."*
> *"Every factory will have an AI factory inside it."*

**核心洞察**：

架構圖漏掉最重要的一層：**本地推理能力**。

現在把所有 AI 呼叫都送到雲端（Claude Max、Gemini）。這在 2026 年還可以，但 2028 年聖保羅搬進 6000 坪工廠時，會遇到三個問題：

1. **延遲**：工廠裡的即時決策等不了 2 秒
2. **頻寬**：MES 每秒幾百筆數據，全上雲太貴
3. **隱私**：客戶訂單、成本結構，真的要送到美國？

**建議**：2027 年就開始規劃 **Edge AI**。在工廠裡放一台本地推理伺服器。簡單的 OCR、異常偵測在本地做，複雜的決策才上雲。這不是省錢的問題，是**能不能做到即時**的問題。

---

### Peter Thiel（PayPal 共同創辦人，《從零到一》作者）

> *"Competition is for losers."*
> *"If your company has no moat, you're just doing free R&D for someone else."*

**核心洞察**：

整份規劃在講「怎麼做」，但沒回答最重要的問題：

**為什麼是你？為什麼鼎新不自己做這個 Agent 智慧層？**

現在設計的所有 Skill，鼎新的工程師都看得懂，都能抄。他們有 ERP 核心，有客戶關係，有資料。你有什麼？

**你需要一個他們不能抄或不願意抄的東西**：
- **產業 know-how**？你比鼎新更懂烘焙業？
- **速度**？你能比他們快 2 年？
- **商業模式**？他們賣授權，你賣訂閱，他們不敢自相殘殺？

想清楚這個，再談技術。
