## 一句話講完這份報告

> 本地推理環境的真正價值，不在於「模型有多強」，而在於能否把**可驗證的自動化**嵌入既有流程——讓人力從「抄、貼、對、找」這些重複勞動中解放，同時確保敏感資料不離開公司網路。

說起來有點諷刺。經濟部 2025 年的調查結果擺在眼前：超過九成的台灣中小企業對 AI 的理解還停留在「聽過、大概知道」的階段。這不是技術問題，是**認知斷層**。

很多人以為「買大主機就等於擁有 AI 能力」，這跟「買了健身器材就會變壯」的邏輯一樣天真。真正讓 AI 落地的關鍵在於：你有沒有把它變成一條可以重複運轉、可以被驗證對錯的生產線。

**洞察**：大企業買 AI 是為了「scale」，中小企業買 AI 應該是為了「省人」。兩者的投資邏輯完全不同。前者追求市場規模，後者追求成本結構。

思銳科技打算推的本地推理環境，本質上是在賣一個承諾：「你的資料不用出公司，但你可以享受 AI 帶來的效率紅利。」這個承諾在醫療、金融、製造這些對資料隱私敏感的產業裡，或許比任何雲端 AI 都來得有吸引力。

## 台灣市場現況

2025 年底的數據給了我們一個很清楚的畫面：台灣中小企業數量創下歷史新高（171.6 萬家），但對 AI 的認知還停在非常初期的階段。這代表什麼？代表**市場還沒被教育完，但需求已經存在**。

經濟部的調查點出了幾個關鍵障礙：「尚無明確應用需求」排第一，接著是「對 AI 不理解」和「導入成本高」。實際上這三件事是同一件事的不同說法——企業主不知道 AI 能幫他做什麼，所以覺得沒需求；不理解，所以覺得貴。

但有意思的是，同一份調查也顯示：15% 的中小企業已經把關鍵營運資料做到「可視化」甚至「可預測」的程度。這 15% 不是不存在，只是還沒找到對的工具。

### 市場痛點

**痛點 #1**：「我知道 AI 很厲害，但不知道能幫我做什麼」——這是最常聽到的話。

**痛點 #2**：「我們的資料很敏感，不敢丟雲端」——尤其是製造業的配方、模具參數。

**痛點 #3**：「用 API 算下來一年要幾十萬，划不來」——尤其是高頻使用場景。

**市場機會**：思銳科技切入的不是「最大的市場」，而是「最痛的市場」。對資料隱私敏感、有高頻 AI 使用需求、但又請不起專職 AI 團隊的中小企業——這個族群的需求是真實存在的，只是還沒被好好滿足。

## 資料主權：為什麼「不出網」這麼重要

本地推理的真正賣點，不是「比雲端便宜」，而是「敏感資料永遠不離開你的控制範圍」。

說起來，很多企業選擇雲端 AI 是因為「方便」。但方便的代價是什麼？你的客戶資料、合約條款、產品配方、財務數字、員工個資——全部都要經過別人的伺服器。就算服務商承諾「不會看、不會存」，**你能證明嗎？出事的時候，誰負責？**

這不是 paranoia，是風險管理的基本功。尤其對製造業的配方參數、金融業的交易資料、醫療業的病歷、法律業的合約——這些東西一旦外洩，損失的不只是錢，還有信任。

### 三大支柱

**資料不出境**：所有運算都在內網完成。原始資料、中間產物、最終結果——全部留在你的機房或辦公室。沒有 API call，沒有雲端 log，沒有第三方能碰到你的資料。

**合規可稽核**：個資法、GDPR、產業特定規範——本地部署讓你能完整掌握資料處理的每一步。稽核來的時候，你可以明確說明資料在哪、誰能存取、怎麼處理。

**供應商中立**：不被單一雲端服務綁定。模型可以換、框架可以改、服務商可以談——但你的資料永遠是你的。這是長期競爭力的基礎。

**製造業**

- 風險：配方外洩 = 競爭優勢歸零

**金融業**
- 客戶交易紀錄、信用評估資料
- 風控模型、反洗錢偵測邏輯
- 投資策略、市場分析報告
- 風險：金管會合規 + 客戶信任崩盤

**醫療業**
- 病歷、診斷紀錄、醫療影像
- 臨床試驗數據、研究資料
- 藥品配方、治療方案優化
- 風險：HIPAA/個資法 + 醫療倫理

**法律業**
- 合約條款、訴訟策略文件
- 客戶機密資訊、盡職調查報告
- 智慧財產權文件、專利分析
- 風險：律師保密義務 + 利益衝突

**政府/國防**
- 公文、政策草案、內部簽呈
- 國防相關技術文件、採購資料
- 人員資料、安全等級分類
- 風險：國家安全 + 資安法規

**研發/科技**
- 專利申請前的技術文件
- 演算法、模型架構、訓練資料
- 競爭分析、市場情報
- 風險：營業秘密 + 先發優勢喪失

### 資料主權驅動的 AI 應用場景

**敏感文件處理**
- 合約審閱：條款比對、風險標註、修改建議——全在內網完成

- 專利搜尋：技術比對、前案分析——保護申請前的秘密

**內部溝通分析**
- 會議紀錄：逐字稿、摘要、待辦——不讓第三方聽到內部討論
- 客訴分析：情緒偵測、問題分類——客戶資料留在內部

**核心系統整合**
- ERP/MES 對接：訂單、庫存、排程——不讓營運資料外流
- CRM 分析：客戶行為、銷售預測——客戶關係是核心資產
- 程式碼庫：自動文件、Bug 修復——原始碼不出公司

**合規自動化**
- PII 遮罩：自動偵測並遮蔽身分證、信用卡、地址
- 文件分級：機密/內部/公開自動標註
- 稽核追蹤：誰存取了什麼、做了什麼改動——完整日誌

## 兩條核心生產線

我們花了很多時間研究這份報告的底層邏輯，得出一個結論：思銳科技要賣的不是「主機」，而是「生產線」。這兩件事差很多。

主機是資產，買了就放在那裡，用不用得起來是你的事。生產線是流程，每個環節都有輸入、有輸出、有驗證機制。企業買的應該是後者。

### 生產線 A：Coding Agent

**技術堆疊**：OpenCode + GLM-4.7 本地推理 + Skills 技能系統

**這條線能做什麼**：
- 新專案接手：自動生成架構圖、模組責任、風險清單
- Bug 修復：讀 log → 寫測試 → 修正 → 跑 CI 直到綠燈
- 大量重構：框架升級、lint 修復、API 調整
- 測試補齊：針對關鍵模組自動補單元測試
- DevOps 代理：Dockerfile、CI pipeline 自動修正
- API 文件：從程式碼自動抽 API、例子、錯誤碼

#### Skills：可傳承的專業技能

Skills 讓通用 Agent 也能具備你公司的專業知識。把業務流程、開發規範、程式碼慣例封裝成 `SKILL.md`，Agent 按需載入，新人也能立即上手。

**Skills 分類**：
- **流程封裝**：部署流程、測試規範、Code Review 標準
- **公司規範**：命名慣例、架構原則、安全準則
- **專案知識**：專案特定邏輯、API 用法、地雷區

**結構**：my-skill/ → SKILL.md（必要）+ scripts/（可執行）+ references/（文件）+ assets/（模板）

**硬性門檻（沒通過不准合）**：
- 必須產出可讀 diff
- 必須跑過 tests
- 必須 lint 通過
- 必須有回滾指令

### 生產線 B：數字人影音工廠

**技術堆疊**：LTX-2 + XTTS + 工作流整合

**這條線能做什麼**：
- 業務個人化影片：同腳本批次產出不同客戶版本
- 客服教學：SOP 變成會講話的教學片
- 內訓短片：制度、資安、流程系列化產出
- 產品上架：商品說明、安裝教學多平台尺寸
- 多語版本：同影片改語音字幕做日/英/東南亞版
- 展場導覽：迎賓/產品亮點數字人循環播放

**技術堆疊**：
- LTX-2 影像生成
- XTTS 語音合成
- Wav2Lip 嘴型同步
- ComfyUI 工作流

**額外選項**：Llama 4 Scout / Maverick — Meta 釋出的原生多模態與超長上下文模型，屬於「可走私有部署」的模型系，適合需要處理超長文件或多模態場景的企業。

## Coding Agent 深度拆解

### 技術定位

**OpenCode**：開源 coding agent（CLI/TUI），內建 **plan（唯讀分析）** 與 **build（可改檔/可跑指令）** 模式；支援把外部工具接成 MCP tools，以及 **Skills 技能系統**

**GLM-4.7（32B active）**：開源權重模型，主打 coding agent 場景的 tool calling 與「多輪保留思路/降低資訊流失」；支援 vLLM / SGLang 本地部署

#### Skills：讓通用 Agent 具備專業技能

Skills 是可重複使用的指令集，讓 Agent 可以按需載入專業知識與工作流程。每個 Skill 是一個資料夾，包含 `SKILL.md` 定義檔。

- **專案層級**：.opencode/skills/* (最高優先)
- **全域層級**：~/.config/opencode/skills/*

**價值**：把業務流程、公司規範、開發慣例封裝成 Skill，新人也能立即上手，知識不再只存在老員工腦中

### 48GB 顯存的價值

- 同時跑 LLM + embedding + reranker + 索引/測試
- 更長上下文：大 repo 掃描、長 log、長規格
- 多工批次：一晚跑完掃 repo → 產文件 → 補測試 → 產 PR

### 中小企業最有用的 10 種 Agent 工作流

1. **新專案接手/讀舊系統**：自動生成架構圖、模組責任、關鍵資料流、風險清單
2. **規格 → 實作**：需求文件拆成 task list、建分支、改程式、補測試、開 PR
3. **Bug 重現 → 修到綠燈**：讀 log/stack trace、寫最小重現測試、反覆修正直到 CI pass
4. **大量重構/框架升級**：Python/Node/Java 版本升級、API 調整、批次 lint/format
5. **自動補齊測試**：關鍵模組補單元測試、產生測資、生成 mock、跑 coverage
6. **DevOps/容器化代理**：Dockerfile、compose、k8s manifests、CI pipeline 修正
7. **資料工程代理**：SQL/ETL pipeline、欄位血緣、資料品質檢查、異常規則
8. **SDK / API 文件自動化**：從程式碼抽 API、例子、錯誤碼、變更摘要
9. **工具串接型任務（MCP）**：Jira/GitHub/GitLab/DB/內網 API 接成工具，agent 直接執行
10. **內網合規開發**：原始碼不出公司；配合稽核紀錄（命令/diff/工具呼叫紀錄）

### 部署路線：plan → build → Skills 分段控風險

1. **Plan 模式**：唯讀分析、掃 repo

## 第一性原理解析

往下挖五層，找到真正的「為什麼」——從「買主機」一路問到「企業為什麼存在」

### 1. 表層需求
「用一台高顯存 GPU 主機把模型推理、工具執行、內容生成做成內網可重複運轉的生產線。」

### 2. 真正在解決的問題
「把企業內部的非結構化訊號（文件、對話、畫面、程式碼）轉成可執行的結構（指令、差異、決策、流程）。」

### 3. 為什麼這件事值錢
「用算力替代稀缺的人類認知資源（注意力、工作記憶、搜尋與比對能力）。48GB 顯存讓代理能同時處理大 repo、長規格、長 log 而不退化。」

### 4. 更底層的物理意義
「把電能轉成對資訊的操作（壓縮、排序、對齊、生成、校驗），降低不確定性帶來的成本——錯誤、延遲、返工、溝通摩擦、版本混亂。」

### ∞ 終極本質
「**生存約束下的能量—資訊交換**：任何系統要延續存在，就必須消耗能量處理資訊以降低不確定性（熵）。本地推理環境，本質上是一種局部熵減的手段。」

講白了，企業買這套系統不是在買「AI」，是在買「秩序」。把混亂的輸入變成可追蹤的輸出，把人腦做不完的比對工作交給機器，然後把省下來的時間拿去做真正需要創造力的事。

## 90 天落地藍圖

很多 AI 專案失敗的原因不是技術不行，是野心太大。想一次解決所有問題，結果什麼都沒解決。這份報告給出的建議是：**先把 2-3 個高頻工作做成可重複的流水線，介面最後再補。**

### Phase 01：第 1-30 天 — 選定戰場

不要什麼都想做，先找最痛的那一個

1. 盤點公司裡「花最多人力在抄、貼、對、找」的工作
2. 從中選出 1 個最高頻、最痛、資料格式最固定的任務
3. 定義「成功長什麼樣」——要有可量化的指標
4. 準備 10-20 個真實案例作為測試資料

### Phase 02：第 31-60 天 — 做出 MVP

先跑通一條線，再想怎麼擴張

1. 部署本地推理服務（GLM-4.7 或 Qwen），確認硬體正常
2. 把選定的任務拆成「輸入 → 處理 → 驗證 → 輸出」四個步驟
3. 每個步驟都要有「怎樣算通過」的檢查機制
4. 用測試資料跑 10 次，記錄成功率與失敗原因

### Phase 03：第 61-90 天 — 上線與迭代

讓它真的跑起來，然後收集回饋

1. 找 2-3 個內部使用者做 beta 測試
2. 建立「回報問題 → 修正 → 重測」的迴圈
3. 記錄實際省下的時間 / 減少的錯誤 / 提升的產出
4. 決定下一個要自動化的任務

## 五個常見的坑

別人踩過的，你就不用再踩一次

**坑 #1：以為買大主機就會變強**
強的是吞吐量，不是智商。品質主要靠任務設計、檢索、驗證。

**坑 #2：以為 Agent 能自動做完所有事**
Agent 只能縮短流程，最後一定要有守門機制——tests、schema、抽樣覆核。

**坑 #3：先做平台 / 介面**
先把 2-3 個高頻工作跑通，介面最後補。順序搞反會浪費很多資源。

**坑 #4：沒驗證就讓 Agent 動產線**
沒有「測試 / 校驗 / 抽樣覆核」，本地或雲端都會變成幻覺生產機。

**坑 #5：追求電影級數字人**
中小企業要的是轉換率與產能，不是電影級寫實。先用模板化短片驗證效果，有需要再升級。

## 最後一句話

> 「本地推理環境的真正考驗，不是**能不能跑起來**，而是**跑起來之後能不能一直跑**。」

這需要的不只是硬體，還有流程設計、驗證機制、持續迭代的決心。思銳科技如果能把這三件事打包成服務，而不只是賣主機，那才是真正的差異化。
